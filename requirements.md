# 要件定義書（初期ドラフト）

> このファイルはドラフトです。実際の業務内容に合わせて内容を追記・修正してください。

## 1. アプリの目的

クレーム内容の要点を入力するだけで、生成AIが以下2種類の報告書文面を自動整形し、最終的に**Word形式（.docx）**でダウンロードできるようにする。

- **社外用報告書**：お客様や取引先向けに提出する、丁寧かつ簡潔な文面
- **社内用報告書**：上長・関係部署への共有を目的とした、経緯・原因・対応状況が分かる詳細な文面

## 2. 背景・課題（想定）

- クレーム対応の都度、報告書をゼロから手作業で作成しており時間がかかる。
- 社外向け・社内向けで求められる文体・情報量が異なるため、二重に文章を作成する手間がある。
- 担当者によって報告書の書き方・粒度にばらつきがある。
- 完成した文書はWord形式で関係者に共有・保存する必要がある。

> ※ここは実際の業務課題に合わせて具体的に書き直してください。

## 3. 利用者（想定ユーザー）

- クレーム対応窓口の担当者（一次受け・記録者）
- 品質管理・カスタマーサポート部門の担当者
- 報告書を確認・承認する上長

## 4. 利用シーン（想定）

1. 担当者がお客様からのクレームを受け付ける。
2. 担当者がアプリに、クレームの要点（発生日、内容、対応状況など）を入力する。
3. アプリ（生成AI）が入力内容をもとに、社外用・社内用の報告書文面を自動生成する。
4. 担当者が生成された文面を確認・必要に応じて修正する。
5. Word形式でファイルを出力し、社内共有・社外提出に利用する。

## 5. 機能要件

### 5.1 入力機能
- クレームの要点を入力するフォーム（詳細は [input_items.md](input_items.md) を参照）
- 入力内容の一時保存（下書き保存）機能（要検討）

### 5.2 生成機能
- 入力内容をもとに、生成AIが以下を自動作成する
  - 社外用報告書の文面
  - 社内用報告書の文面
- 生成結果をアプリ画面上でプレビュー・編集できる

### 5.3 出力機能
- 生成・編集した文面を **Word形式（.docx）** ファイルとして出力する
- 出力ファイル名のルール化（例：`YYYYMMDD_クレーム報告書_社外用.docx` など）（要検討）

### 5.4 その他（検討中）
- 過去の入力・生成履歴を保存し、再利用できるようにするか
- 複数人での確認・承認フロー（ワークフロー機能）を持たせるか
- 社内テンプレート（決まった書式）に沿った出力にするか

## 6. 非機能要件（検討中）

| 項目 | 内容（ドラフト） |
|---|---|
| セキュリティ | クレーム内容には個人情報・機微情報が含まれる可能性があるため、取り扱いに注意する。外部の生成AIサービスを利用する場合、データの送信範囲・保存ポリシーを確認する |
| パフォーマンス | 報告書生成は数秒〜数十秒程度で完了することが望ましい |
| 可用性 | 社内利用が中心のため、常時稼働（24/365）までは求めない想定 |
| 対象環境 | PCブラウザでの利用を想定（詳細は要検討） |
| 想定利用人数 | 数名〜十数名規模（要確認） |

## 7. 使用する生成AI・技術スタック

開発手順書（[development_procedure.md](development_procedure.md)）Phase 1にて、以下のとおり暫定決定した（詳細は [CLAUDE.md](CLAUDE.md) の「技術スタック」欄を参照）。

- アプリ形態：Streamlit（Python）による社内向けWebアプリ
- 生成AI：Anthropic Claude API　※現時点はプロトタイプ・テスト採用。本番利用の可否は9章の未確定事項として引き続き検討する
- Word出力方法：`python-docx` ライブラリ

## 8. スコープ外（今回は対応しない事項）

- クレーム管理システム（案件のステータス管理・検索等）としての機能
- 他システム（顧客管理システム等）との自動連携
- 多言語対応（まずは日本語のみを想定）
